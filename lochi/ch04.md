# MySQL 엔진 아키텍처

## MySQL 전체 구조

![[Pasted image 20260504072450.png|627]]

### MySQL 엔진

- DBMS의 두뇌에 해당하는 처리 수행
- MySQL 서버에 하나만 존재
- 구성
  - 커넥션 핸들러: 클라이언트로부터의 접속 및 쿼리 요청 처리
  - SQL 파서
  - 전처리기
  - 옵티마이저: 쿼리 최적화 실행

### 스토리지 엔진

- 실제 데이터를 디스크 스토리지에 저장하거나, 데이터를 읽어오는 담당
- MySQL 엔진과 다르게 여러 개를 동시에 사용할 수 있음

  ```sql
  CREATE TABLE test_table (fd1 INT, fd2 INT) ENGINE=INNODB;
  ```

> [!question]- **_왜 스토리지 엔진을 여러 개 쓸 수 있게 했을까?_**
>
> - 테이블 마다 요구사항이 다르기 때문
> - 어떤 엔진은 조회에 강하고, 어떤 엔진은 쓰기/트랜잭션에 강함
> - 성능 외에도 기능, 데이터 성격, 운영 목적 등에 따라서 여러 개를 사용할 수 있음

> [!question]- **_그럼 스토리지 엔진 별로 어떤 차이가 있을까?_**
>
> | 테이블 성격                | 적합한 엔진 예시 | 이유                          |
> | -------------------------- | ---------------- | ----------------------------- |
> | 주문, 결제, 회원 정보      | InnoDB           | 트랜잭션, 복구, 동시성에 강함 |
> | 임시 계산용 테이블         | MEMORY           | 메모리에 저장해서 빠름        |
> | CSV 파일처럼 다루는 테이블 | CSV              | 데이터를 CSV 형식으로 저장    |
> | 로그/아카이브 데이터       | ARCHIVE          | 압축 저장에 유리              |
> | 예전 읽기 위주 테이블      | MyISAM           | 단순 조회 성능 위주           |

> [!question]- **_옛날 건 구린거 아닌가?_**
>
> - 조금 부족한 점은 많지만, 설계 목적이 달라서 목적성에 맞는 엔진을 사용하면 됨
> - 실무 기준에서는 일반적인 테이블은 거의 무조건 InnoDB를 사용하면 됨

### 핸들러 API

- 핸들러 요청: MySQL 엔진의 쿼리 실행기에서 데이터를 쓰거나 읽어야 할 때의 각 스토리지 엔진에 쓰기 또는 읽기 요청
- 핸들러 API: 핸들러 요청에 사용되는 API
- `SHOW GLOBAL STATUS LIKE 'Handler%';` : 핸들러 API를 통해 얼마나 많은 데이터 작업이 있었는지 확인

## MySQL 스레딩 구조

![[Pasted image 20260504072558.png]]

- MySQL 서버는 프로세스 기반이 아니라 스레드 기반으로 작동함
- 크게 포그라운드(Foreground) 스레드와 백그라운드(Background) 스레드로 구분

### 포그라운드 스레드(클라이언트 스레드)

- MySQL 서버에 접속된 클라이언트 수만큼 존재
- 각 클라이언트 사용자가 요청하는 쿼리 문장 처리
- 클라이언트가 커넥션을 종료하면 해당 커넥션을 담당하던 스레드는 다시 스레드 캐시(Thread cache)로 되돌아감
- 이미 캐시에 일정 개수 이상의 대기 중인 스레드가 있으면, 스레드 캐시에 넣지 않고 스레드를 종료 시켜 개수 유지
  - `thread_cache_size` 시스템 변수로 최대 개수 설정

- 포그라운드 스레드는 데이터를 MySQL의 데이터 버퍼나 캐시로부터 가저옴
- 버퍼나 캐시에 없는 경우에는 직접 디스크 데이터 or 인덱스 파일로부터 데이터를 읽어와서 작업을 처리함
- MyISAM 테이블은 디스크 쓰기 작업까지 포그라운드 스레드가 처리하지만, InnoDB 테이블은 데이터 버퍼가 캐시까지만 포그라운드 스레드가 처리하고, 나머지 버퍼로부터 디스크까지 기록하는 작업은 백그라운드 스레드가 처리

> [!question]- **_MyISAM 테이블?_**
>
> - MySQL 스토리지 엔진 중 하나인 ‘MyISAM 스토리지 엔진’으로 생성된 테이블
> - 예전 MySQL에서 기본으로 많이 사용됨
> - MyISAM 특징
>   - 트랜잭션 미지원
>   - 외래 키 미지원
>   - 테이블 단위 락
>   - 읽기 성능 우수
>   - 복구 취약
>   - 파일 기반 저장 등

> [!question]- **_InnoDB 테이블은 ‘InnoDB 스토리지 엔진’으로 생성된 테이블인가?_**
>
> - 맞음
> - 현재 가장 일반적으로 많이 쓰이는 방식
> - InnoDB 특징: 트랜잭션 지원, 외래 키 지원, 행 단위 락, 장애 복구 강함, 동시성 우수, 클러스터링 인덱스 사용

### 백그라운드 스레드

- InnoDB 기준
  - Insert Buffer 를 병합하는 스레드
  - 로그를 디스크로 기록하는 스레드 (중요)
  - InnoDB 버퍼 풀의 데이터를 디스크에 기록하는 스레드
  - 데이터를 버퍼로 읽어 오는 스레드 (중요)
  - 락이나 데드락을 모니터링 하는 스레드

- MySQL 5.5 부터 읽기/쓰기 스레드 개수를 2개 이상 지정할 수 있게 됨
- InnoDB에서도 읽는 작업은 주로 클라이언트 스레드에서 처리되기 때문에 읽기 스레드는 많이 설정할 필요가 없음
- 쓰기 스레드는 아주 많은 작업을 백그라운드로 처리하기 때문에, 일반적인 내장 디스크를 사용할 때는 2~4, DAS나 SAN 같은 스토리지를 사용할 때는 디스크를 최적으로 사용할 수 있을 만큼 충분히 설정하는 것이 좋다.

> [!question]- **_DAS, SAN 스토리지는 일반 디스크와 무슨 차이가 있길래 스레드를 충분히 설정해줘야 할까?_**
>
> - DAS나 SAN은 일반적인 단일 내장 디스크보다 동시에 처리할 수 있는 I/O 용량이 훨씬 크기 때문에, InnoDB 쓰기 스레드를 더 많이 설정해야 그 성능을 충분히 끌어낼 수 있습니다.
> - 추천 시작 기준 (처음부터 크게 잡지 않아야 함)
>
> | 스토리지 환경  | 권장 시작값 |
> | -------------- | ----------- |
> | 단일 HDD       | 2           |
> | 단일 SATA SSD  | 4           |
> | 단일 NVMe SSD  | 4~8         |
> | RAID SSD / DAS | 8~16        |
> | 고성능 SAN     | 8~32        |

- “데이터의 쓰기 작업은 지연되어 처리될 수 있지만, 데이터의 읽기 작업은 절대 지연이 될 수 없다.”
- 그래서 상용 DBMS에는 쓰기 작업은 배치 처리 기능이 탑재 되어 있음
  - MyISAM은 그렇지 않고, 사용자 스레드가 쓰기 작업까지 함께 처리하도록 설계되어 있음

> [!QUESTION]- **_[질문] 포그라운드 스레드와 백그라운드 스레드_**
> 포그라운드 스레드
>
> - 사용자의 요청을 직접 처리하는 스레드
>   - 클라이언트 요청 받기
>   - SQL 파싱
>   - 권환 확인
>   - 옵티마이저 실행
>   - 실행 계획 수정
>   - 스토리지 엔진에 데이터 읽기 요청
>   - 결과를 클라이언트에게 반환
>
> 백그라운드 스레드
>
> - 사용자가 직접 기다리지 않아도 되는 내부 작업을 뒤에서 처리하는 스레드
>   - 변경된 데이터를 나중에 디스크에 기록
>   - redo log를 디스크에 기록
>   - buffer pool 정리
>   - dirty page flush
>   - undo/purge 정리
>   - change buffer 병합
>   - deadlock 감시
>
> MySQL 엔진 = 주로 포그라운드 스레드가 실행
> 스토리지 엔진 = 포그라운드 스레드가 호출해서 데이터 읽기/쓰기 수행
> InnoDB 내부 후처리 = 백그라운드 스레드가 수행

> [!QUESTION]- **_읽기 쿼리를 처리하는 과정_**
> 예시 쿼리
>
> ```SQL
> SELECT name
> FROM users
> WHERE id = 10;
> ```
>
> 처리 과정 (대부분 포그라운드)
>
> 1. 클라이언트가 쿼리 보냄
> 2. 포그라운드 스레드가 쿼리를 받음
> 3. MySQL 엔진이 SQL 문장을 분석함
>    1. SQL을 MySQL이 이해할 수 있는 구조로 바꿈
>    2. 문법 오류가 있으면 바로 에러 발생
> 4. 전처리기가 테이블/컬럼/권한을 확인함
>    1. users테이블이 실제로 있는가?
>    2. name 컬럼이 실제로 있는가?
>    3. id 컬럼이 실제로 있는가?
>    4. 이 사용자가 users 테이블을 SELECT 할 권한이 있는가
> 5. 옵티마이저가 실행 방법을 고르고 계획을 세움
> 6. 실행 엔진이 계획대로 실행함
> 7. MySQL 엔진이 스토리지 엔진에게 데이터 읽기를 요청함
> 8. InnoDB가 데이터를 찾음
>    1. 우선 메모리인 버퍼 풀을 확인하고 있으면 가져옴
>    2. 없으면 해당 데이터 페이지를 읽어와 버퍼 풀에 올린 뒤 row를 찾음 (백그라운드)
> 9. MySQL 엔진이 결과를 가공함
> 10. 포그라운드 스레드가 클라이언트에게 결과를 반환함

> [!QUESTION]- **_쓰기 쿼리를 처리하는 과정_**
> 예시 쿼리
>
> ```SQL
> UPDATE users
> SET name = 'Lee'
> WHERE id = 10;
> ```
>
> 앞부분은 읽기 처리 과정과 거의 동일
>
> ```
> 클라이언트 요청
> → 포그라운드 스레드가 받음
> → 파싱
> → 전처리
> → 옵티마이징
> → 실행
> → InnoDB에 row 변경 요청
> ```
>
> InnoDB는 해당 row를 찾아서 버퍼 풀의 데이터를 변경함
> 하지만, 변경된 데이터를 매번 즉시 디스크에 완전히 써버리진 않음.
> 메모리에 먼저 반영하고, redo log 등을 이용해 장애 복구 가능성을 확보한 뒤, 실제 데이터 파일 반영은 나중에 백그라운드 스레드가 처리
>
> 포그라운드 스레드:
>
> - 사용자 UPDATE 처리
> - 버퍼 풀 변경
> - 필요한 로그 처리
> - 클라이언트에게 완료 응답
>
> 백그라운드 스레드:
>
> - 나중에 dirty page를 디스크에 flush
> - purge 작업
> - change buffer 병합
> - checkpoint 관련 작업

## 메모리 할당 및 사용 구조

![[Pasted image 20260504072633.png|590]]

- MySQL에서 사용되는 메모리 공간은 크게 ‘글로벌 메모리 영역’과 ‘로컬 메모리 영역’으로 구분 가능
- 글로벌 메모리 영역의 모든 메모리 공간은 MySQL 서버가 시작되면서 운영체제로부터 할당
  - (운영체제 종류에 따라 다르지만) 메모리 공간 100%도 가능하고 예약 할당도 가능함

### 글로벌 메모리 영역

- 일반적으로 클라이언트 수와 무관하게 하나의 메모리 공간만 할당됨
  - 필요에 따라 2개 이상의 메모리 공간도 할당받을 수 있지만, 클라이언트 스레드 수와는 무관
  - 생성된 글로벌 영역이 N개라 하더라도 모든 스레드에 의해 공유됨

- 대표적인 글로벌 메모리 영역
  - 테이블 캐시
  - InnoDB 버퍼 풀
  - InnoDB 어댑티브 해시 인덱스
  - InnoDB redo log 버퍼

> [!question]- **_테이블 캐시? InnoDB 버퍼 풀? InnoDB 어댑티브 해시 인덱스? InnoDB redo log버퍼?_**
>
> | 영역                            | 하는 일                                                                                        |
> | ------------------------------- | ---------------------------------------------------------------------------------------------- |
> | **테이블 캐시**                 | 자주 사용하는 테이블의 파일 핸들러/메타정보를 메모리에 보관해서 테이블을 열고 닫는 비용을 줄임 |
> | **InnoDB 버퍼 풀**              | InnoDB 테이블의 데이터 페이지와 인덱스 페이지를 메모리에 캐싱해서 디스크 I/O를 줄임            |
> | **InnoDB 어댑티브 해시 인덱스** | 자주 접근되는 인덱스 페이지에 대해 InnoDB가 자동으로 해시 인덱스를 만들어 검색을 더 빠르게 함  |
> | **InnoDB redo log 버퍼**        | 변경된 데이터를 디스크에 바로 쓰기 전에 리두 로그 내용을 잠시 모아두는 메모리 공간             |

### 로컬 메모리 영역

- 로컬 메모리 영역 = 세션 메모리 영역
- MySQL 서버상에 존재하는 클라이언트 스레드가 쿼리를 처리하는 데 사용하는 메모리 영역
- 클라이언트가 MySQL 서버에 접속하면 클라이언트 요청 처리를 위해 하나씩 할당하게 되는데, 클라이언트 스레드가 사용하는 메모리 공간이라고 해서 클라이언트 메모리 영역이라고도 함.
  - 클라이언트와 MySQL 서버와의 커넥션을 세션이라고 하기 때문에 세션 메모리 영역이라고도 표현

- 각 클라이언트 스레드별로 독립적으로 할당되며, 절대 공유되지 않음
- 최악의 경우 MySQL 서버가 메모리 부족으로 멈춰 버릴 수도 있으므로, 적절한 메모리 공간을 설정해줘야 함
- 각 쿼리의 용도별로 필요할 때만 공간이 할당되고 필요하지 않은 경우에는 MySQL이 메모리 공간을 할당조차도 하지 않을 수도 있음 (쿼리를 실행하는 순간에만 할당했다가 다시 해제)
  - 대표적으로 소트 버퍼나 조인 버퍼 같은 공간

- 커넥션이 열려 있는 동안 계속 할당된 상태로 남아있는 공간도 있음
  - 커넥션 버퍼, 결과 버퍼

- 대표 로컬 메모리 영역
  - 정렬 버퍼
  - 조인 버퍼
  - 바이너리 로그 캐시
  - 네트워크 버퍼

> | 영역               | 하는 일                                                                                |

> | 정렬 버퍼          | `ORDER BY`, `GROUP BY` 같은 정렬 작업을 할 때 사용하는 메모리 공간                     |
> | 조인 버퍼          | 인덱스를 효율적으로 못 쓰는 조인에서 임시로 데이터를 담아 조인 처리를 돕는 메모리 공간 |
> | 바이너리 로그 캐시 | 트랜잭션 중 발생한 변경 내용을 바이너리 로그에 쓰기 전 잠시 저장하는 메모리 공간       |
> | 네트워크 버퍼      | 클라이언트와 MySQL 서버가 데이터를 주고받을 때 사용하는 송수신 버퍼                    |

## 플러그인 스토리지 엔진 모델

![[Pasted image 20260504072713.png]]

- 스토리지 엔진 뿐만 아니라 전문 검색 엔진을 위한 검색어 파서, 사용자 인증을 위한 Native Authentication, Caching SHA-2 Authentication 등이 모두 플러그인으로 구현되어 제공됨
- MySQL은 기본적으로 많은 스토리지 엔진을 가지고 있지만, 사용자의 요구 조건을 만족시키기 위해 기본적으로 제공하는 스토리지 엔진 이외에 부가적인 기능을 더 제공하는 스토리지 엔진이 필요할 수도 있음
  - 그래서 전문 회사나 사용자가 직접 스토리지 엔진을 개발하는 것도 가능함

- [MySQL 쿼리 실행 과정] ![[Pasted image 20260504072724.png]]
  - 거의 대부분의 작업이 MySQL 엔진에서 처리되고, 마지막 ‘데이터 읽기/쓰기’ 작업만 스토리지 엔진에 의해 처리됨
    ⇒ 만약 사용자가 새로운 용도의 스토리지 엔진을 만든다고 해도, DBMS의 전체 기능이 아닌 일부분의 기능만 수행하는 엔진을 작성하게 됨.

> [!question]- **_위 예시 말고 실제로 사용할만한 예시 플러그인 스토리지 엔진에는 뭐가 있는지?_**
>
> - 주로 특수한 저장 목적이 있을 때 DB 회사나 인프라 팀이 만듦
>
> | 목적             | 만드는 스토리지 엔진 예시                          |
> | ---------------- | -------------------------------------------------- |
> | 쓰기 성능 최적화 | LSM-tree 기반 엔진                                 |
> | 압축률 개선      | 압축 저장에 특화된 엔진                            |
> | 분석 쿼리 최적화 | 컬럼 기반 저장 엔진                                |
> | 분산 저장        | 샤딩/클러스터링 엔진                               |
> | 외부 데이터 접근 | 원격 DB, 파일, 오브젝트 스토리지를 테이블처럼 접근 |
> | 테스트/개발용    | 실제 저장은 안 하고 엔진 인터페이스 실험           |

- MySQL에서의 ‘핸들러(handler)’: MySQL 엔진이 각 스토리지 엔진에게 데이터를 읽어오거나 저장하도록 명령하려면 반드시 핸들러를 통해야 함
- 하나의 쿼리는 여러 하위 작업으로 나뉘는데, 각 하위 작업이 MySQL 엔진 영역에서 처리되는지 아니면 스토리지 엔진 영역에서 처리되는지 구분할 줄 알아야 함.

## 쿼리 실행 구조

![[Pasted image 20260504072805.png]]

### 쿼리 파서

- 사용자 요청으로 들어온 쿼리 문장을 토큰(MySQL이 인식할 수 있는 최소 단위나 어휘나 기호)으로 분리해 트리 형태의 구조로 만들어 내는 작업

> [!question]- **_왜 트리 형태의 구조로 만들어 낼까?_**
>
> SQL 문장이 단순한 토큰 나열이 아니라, 절·표현식·서브쿼리·조건 등이 서로 포함 관계를 갖는 계층 구조이기 때문에
>
> 1. SQL의 포함 관계를 보존하기 위해
>    - 조건식, 함수 호출, 서브 쿼리, 괄호로 묵인 쿼리 표현 등은 내부에 다시 하위 표현식을 가짐
>    - 트리로 ‘부모-자식’관계를 그대로 표현할 수 있음
> 2. 의미 분석을 하기 위해
> 3. 최적화와 변환을 하기 위해
>    - 중복 claues, 상수값 제거, semi-join transformation 등 적용
>    - 최적화 과정에서 변환 가능한 내부 구조임
> 4. 실행 계획을 만들기 위해
>
> - 결론: 토큰의 순서를 보존하는 것만으로는 SQL의 문법적, 의미적 구조를 충분히 표현할 수 없기 때문
> - 쿼리 예시
>
> ```sql
> SELECT name
> FROM users
> WHERE age > 20 AND city = 'Seoul';
> ```
>
> - 트리 구조 변환시
>
> ```
> SELECT_STATEMENT
> ├── SELECT_LIST
> │   └── COLUMN_REFERENCE
> │       └── name
> ├── FROM_CLAUSE
> │   └── TABLE_REFERENCE
> │       └── users
> └── WHERE_CLAUSE
>     └── BOOLEAN_EXPRESSION
>         └── AND
>             ├── COMPARISON_EXPRESSION
>             │   ├── COLUMN_REFERENCE
>             │   │   └── age
>             │   ├── OPERATOR
>             │   │   └── >
>             │   └── LITERAL
>             │       └── 20
>             └── COMPARISON_EXPRESSION
>                 ├── COLUMN_REFERENCE
>                 │   └── city
>                 ├── OPERATOR
>                 │   └── =
>                 └── LITERAL
>                     └── 'Seoul'
> ```

- 쿼리 문장의 기본 문법 오류는 이 과정에서 발견되고 사용자에게 오류 메시지를 전달하게 됨

### 전처리기

- 파서 과정에서 만들어진 파서 트리를 기반으로 쿼리 문장에 구조적 문제가 있는지 체크
- 각 토큰을 테이블 이름, 칼럼 이름, 내장 함수 같은 개체를 매핑해, 해당 객체의 존재 여부와 객체의 접근 권한 등을 확인하는 과정을 이 단계에서 수행함
- 실제 존재하지 않거나, 권한상 사용할 수 없는 개체의 토큰은 이 단계에서 걸러짐

### 옵티마이저

- 사용자 요청으로 들어온 쿼리 문장을 저렴한 비용으로 가장 빠르게 처리할지를 결정하는 역할 담당
- DBMS의 두뇌 역할

> [!question]- **_옵티마이저가 실제로는 더 느린 쿼리로 변환할 가능성이 있지 않을까?_**
>
> 비용 추정·통계·제약 조건 때문에 실제 실행에서는 더 느린 실행 계획이 선택될 수 있다.
>
> - 옵티마이저가 실행 계획을 만들 때 여러 연산의 비용 추정값에 기반한 cost model을 사용함
> - 실제로 모든 계획을 실행해보고 고르는 것이 아니라, 추정 비용을 바탕으로 계획 세움
> - 이 추정이 실제 데이터 분포나 실행 환경과 어긋나면 더 느린 계획이 선택될 수 있음

### 실행 엔진

- 옵티마이저가 회사 경영진이면, 실행 엔진은 중간 관리자, 핸들러는 각 업무의 실무자에 해당
- 만들어진 게획대로 각 핸들러에게 요청해서 받은 결과를 똗 다른 핸들러의 요청의 입력으로 연결하는 역할

### 핸들러 (스토리지 엔진)

- MySQL 서버 가장 밑단에서 MySQL 실행 엔진의 요청에 따라 데이터를 디스크로 저장하고 디스크로부터 읽어 오는 역할을 담당
- 핸들러 == 스토리지 엔진
- MyISAM 테이블을 조작하는 경우에는 핸들러가 MyISAM 스토리지 엔진이 되고,
  InnoDB 테이블을 조작하는 경우에는 핸들러가 InnoDB 스토리지 엔진이 됨

## 복제(Replication)

- 매우 중요한 역할 (16장에서 확인)

> [!question]- **_복제 아주 간단하게 찍먹_**
>
> > 한 MySQL 서버의 데이터를 다른 MySQL 서버들로 복제하는 기능
>
> - `source`: 데이터를 제공하는 서버
> - `replica`: 복제 데이터를 받는 서버
> - 기본적으로 비동기 방식으로 동작
> - replica가 source에 항상 연결되어 있지 않아도 나중에 업데이트 받을 수 있음
>
> ### 동작 원리
>
> - source에서 트랜잭션이 실행, 커밋되고, 그 변경 내용이 replica로 전달되어 적용되는 방식
>
> ### 주요 목적
>
> - 읽기 부하 분산
>   - 쓰기는 source에서, 읽기는 여러 replica에서
> - 백업 및 데이터 복사
>   - source 서버의 데이터를 replica에 복제해 두면 백업, 리포팅, 분석용 서버로 활용 가능
> - 고가용성 구성의 기반
>   - source 장앵 시, replica를 새로운 source로 승격하는 방식의 failover 구성잉 가능함

## 쿼리 캐시 (Query Cache)

- SQL 실행 결과를 메모리에 캐시하고, 동일 SQL 쿼리가 실행되면 테이블을 읽지 않고 즉시 결과를 반환하여 매우 빠른 성능 제공
- 테이블의 데이터가 변경되면 캐시에 저장된 결과 중에서 변경된 테이블과 관련된 것들은 모두 삭제(Invalidate) 해야함 → 심각한 동시 처리 성능 저하 유발 + 여러 버그의 원인 (MySQL 8.0 이전)
- 실제 성능 향상이 거의 없고 여러 버그의 원인으로 지목되어, MySQL 8.0부터는 쿼리 캐시 기능이 완전 제거되고, 시스템 변수도 모두 제거함

## 스레드 풀 (Thread Pool)

- 기본 MySQL 서버 엔터프라이즈 에디션만 제공, Percona Server는 플러그인 형태로 동작해서 커뮤니티 에디션에서도 사용 가능함
- 스레드풀은 사용자의 요청을 처리하는 스레드 개수를 줄여서, 동시 처리되는 요청이 많다 하더라도 MySQL 서버의 CPU가 제한된 개수의 스레드 처리에만 집중할 수 있게 해서 서버의 자원 소모를 줄이는 것이 목적

> [!question]- **_사용자의 요청을 처리하는 스레드 개수를 줄이는데, 왜 속도가 더 향상되지?_**
>
> > 스레드 풀은 동시 실행 스레드 수를 줄여 컨텍스트 스위칭·CPU 캐시 낭비·락 경합을 낮추기 때문에, 요청이 많은 상황에서 오히려 처리 속도와 처리량이 향상될 수 있습니다.
>
> “스레드가 많을수록 항상 더 빠르다”가 아님
>
> - 너무 많은 스레드가 동시에 실행되면 CPU·OS 스케줄러·InnoDB 내부 락 경합에 오히려 부담이 생김
> - MySQL은 **one thread per connection**, 즉 클라이언트 연결 하나당 스레드 하나를 사용하는 방식
> - 그런데 접속 클라이언트가 늘어나고 statement 실행이 많아지면 전체 성능이 저하될 수 있으며, Thread Pool은 이런 오버헤드를 줄이고 성능을 개선하기 위한 대체 스레드 처리 모델
>
> ### 핵심 이유
>
> 1. **컨텍스트 스위칭 오버헤드 감소**
>
> - 너무 많은 스레드가 병렬로 실행되면 context switching overhead가 높아지고, OS scheduler에도 부담
> - Thread Pool은 MySQL 서버 내부의 병렬 실행 수준을 서버가 감당할 수 있는 정도로 제한
> - 즉, 요청은 많아도 “실제로 동시에 CPU를 차지하려는 스레드 수”를 조절해 CPU가 불필요한 스케줄링에 시간을 덜 쓰게 함
>
> 2. **CPU cache 사용 효율 향상 가능**
>
> - 너무 많은 thread stack이 있으면 고병렬 실행 workload에서 CPU cache가 거의 쓸모없게 될 수 있고, Thread Pool은 thread stack reuse를 통해 CPU cache footprint를 줄임.
> - 또한 가장 active한 thread를 재사용해 CPU cache를 더 잘 활용
>
> 3. **InnoDB 내부 경합이 감소**
>
> - 너무 많은 transaction이 동시에 실행되면 resource contention이 증가하고, InnoDB에서는 central mutex를 잡고 있는 시간이 증가함
> - Thread Pool은 너무 많은 transaction이 동시에 시작되지 않도록 제어
>
> ### 주의
>
> **항상 빨라지는 기능은 아님**
>
> - (Percona 공식 문서) 20,000개 미만의 connection에서는 Thread Pool이 큰 이점을 제공하지 않으며, 기본 방식을 유지하는 것이 낫다.
> - (MySQL 공식 문서) Thread Pool 성능을 위해 `thread_pool_size`, `thread_pool_query_threads_per_group`, `thread_pool_max_transactions_limit` 같은 값을 workload에 맞게 조정해야 한다

- 스레드 풀 설치만 한다고 성능이 2배 증가하진 않음
- 동시에 실행 중인 스레드들을 CPU가 최대한 잘 처리해낼 수 있는 수준으로 줄여서 빨리 처리하게 하는 기능이기 때문에, 스케줄링 과정에서 CPU 시간을 제대로 확보하지 못하는 경우 쿼리 처리가 더 느려질 수 있음
- 제한된 수의 스레드만으로 CPU가 처리하도록 적절히 유도한다면 CPU의 프로세서 친화도도 높이고, 운영체제 입장에서는 불필요한 컨텍스트 스위치를 줄여서 오버헤드를 낮출 수 있음  
  <br>
- Percona Server의 스레드 풀은 기본적으로 CPU 코어의 개수만큼 스레드 그룹을 생성하는데, `thread_pool_size` 시스템 변수를 변경해서 조정할 수 있음
  - 일반적으로 CPU 코어의 개수와 맞추는 것이 CPU 프로세서 친화도를 높이는 데 좋음
- MySQL 서버가 처리해야 할 요청이 생기면 스레드 풀로 처리를 이관하는데, 만약 스레드 풀이 처리 중인 작업이 있는 경우에는 `thread_pool_oversubscribe` 시스템 변수(기본값: 3)에 설정된 개수만큼 추가도 더 받아들여서 처리함 - 이 값이 너무 크면 스케줄링해야 할 스레드가 많아져서 스레드 풀이 비효율적으로 작동할 수도 있음
  <br>
- 스레드의 모든 그룹의 스레드가 일을 처리하고 있다면, 스레드 풀은 해당 스레드 그룹에 새로운 작업 스레드(Worker thread)를 추가할지, 아니면 기존 작업 스레드가 처리를 완료할 때까지 기다릴지 여부를 판단
- 스레드 풀의 타이머 스레드는 주기적으로 수레드 그룹의 상태를 체크해서 `thread_pool_stall_limit`시스템 변수에 정의된 밀리초만큼 작업 스레드가 해당 작업을 못 끝내면 새로운 스레드를 생성해서 스레드 그룹에 추가함
  - 스레드 풀의 전체 스레드 개수는 `thread_pool_max_threads` 개수를 넘진 못함
- 모든 스레드가 각자 작업을 처리하는 동안 새로운 쿼리 요청이 들어오더라도 `thread_pool_stall_limit`만큼 기다려야만 새로 들어온 요청을 처리할 수 있음
  - 따라서 응답 시간에 아주 민감한 서비스라면 `thread_pool_stall_limit`시스템 변수를 적절히 낮춰서 설정해야 함
  - 그렇다고 0에 가까운 값으로 설정하는 건 권장 X. 차라리 스레드 풀을 사용하지 말 것
