# 트랜잭션 격리 수준 구현하기

## 1. 트랜잭션 격리 수준이란?

트랜잭션 격리 수준은 **여러 트랜잭션이 동시에 실행될 때, 한 트랜잭션이 다른 트랜잭션의 변경 결과를 어느 시점부터 읽을 수 있는지**를 정하는 설정이다.

DBMS는 여러 트랜잭션을 동시에 실행하면서도 **데이터 정합성**을 지키기 위해 **Lock, MVCC, Snapshot, Undo Log**와 같은 다양한 동시성 제어 기법을 사용한다.

트랜잭션 격리 수준은 이러한 동시성 제어 중에서도, **트랜잭션 간 읽기 일관성을 어느 정도까지 보장할 것인지**를 사용자가 선택할 수 있도록 제공하는 설정이다.

## 2. 분리 기준

**SQL** 표준에서는 다음 **이상 현상을** **어디까지 방지**하는지를 기준으로 격리 수준을 분리한다.

- **Dirty Read**
    - 다른 트랜잭션이 **아직 커밋하지 않은 데이터**를 조회하는 이상 현상
- **Non-repeatable Read**
    - 하나의 트랜잭션 내에서 같은 **조건으로 여러 번 조회**할 때, **결과가 달라지는** 이상 현상
- **Phantom Read**
    - 하나의 트랜잭션 내에서 **같은 조건으로 여러 번 조회**할 때, **처음에는 없던 데이터**가 나타나는 이상 현상

## 3. 트랜잭션 격리 수준 유형

**SQL 표준**에서 정의하는 트랜잭션 격리 수준은 다음 네 가지이다.

### Read Uncommitted

- 다른 트랜잭션이 **아직 커밋하지 않은 데이터도** 읽을 수 있는 격리 수준이다.
    - 즉 **Dirty Read**를 허용한다.
- **가장 낮은** 격리 수준이다.

| 이상 현상 | 발생 여부 |
| --- | --- |
| Dirty Read | 발생 가능 |
| Non-repeatable Read | 발생 가능 |
| Phantom Read | 발생 가능 |

### Read Committed

- **커밋된 데이터**만 읽을 수 있는 격리 수준이다.
    - 따라서 **Dirty Read**는 막을 수 있다.
    - 하지만 같은 트랜잭션 안에서 같은 데이터를 다시 읽을 때, 그 사이 다른 트랜잭션이 커밋했다면 다른 값을 읽을 수 있다.
- **대부분의 DBMS**에서 기본적으로 제공하는 격리 수준이다.

| 이상 현상 | 발생 여부 |
| --- | --- |
| Dirty Read | 방지 |
| Non-repeatable Read | 발생 가능 |
| Phantom Read | 발생 가능 |

#### Repeatable Read

- 하나의 트랜잭션 안에서 같**은 데이터를 반복해서 읽어도 같은 결과를 보장**하는 격리 수준이다.
    - 따라서 **Dirty Read**와 **Non-repeatable Read**를 막을 수 있다.
- 다만 **Phantom Read**를 막는지는 DB의 구현 방식에 따라 다를 수 있다.
    - 예를 들어 **MySQL InnoDB**는 Repeatable Read 격리 수준이지만,  일반 `SELECT`는 트랜잭션 안에서 첫 번째 읽기 시점의 스냅샷을 기준으로 일관된 결과를 읽는다.
- **MySQL**은 **Repeatable Read** 수준의 격리 수준을 기본으로 제공한다.

| 이상 현상 | 발생 여부 |
| --- | --- |
| Dirty Read | 방지 |
| Non-repeatable Read | 방지 |
| Phantom Read | DB 구현에 따라 다름 |

#### Serializable

- 여러 트랜잭션이 동시에 실행되더라도, **트랜잭션들을 하나씩 순서대로 실행한 것**과 결과가 같아야 하는 격리 수준이다.
- **가장 강한** 격리 수준이다.

| 이상 현상 | 발생 여부 |
| --- | --- |
| Dirty Read | 방지 |
| Non-repeatable Read | 방지 |
| Phantom Read | 방지 |

## 4. 트랜잭션 격리 수준 구현하기

트랜잭션 격리 수준은 **개념적인 정책**이고, **DB**는 이를 구현하기 위해 **여러 동시성 제어 기법**을 사용한다.

대표적으로 **H2**, **PostgreSQL**, **MYSQL InnoDB**의 구현방식은 다음과 같다.

### H2

H2는 **MVStore 기반 MVCC**와 **Lock**을 조합해 트랜잭션 동시성을 제어한다.

**MVStore**는 데이터의 여러 버전을 관리하여, 각 트랜잭션이 격리 수준에 맞는 버전을 읽을 수 있도록 한다.

반면 **Lock**은 동시에 변경되면 안 되는 작업을 제어하여, 같은 데이터에 대한 쓰기 충돌을 방지한다.

격리 수준별 동작은 다음과 같이 이해할 수 있다.

- **Read Committed**
    - 조회 시점에 **커밋된 최신 버전**을 읽는다.
    - 따라서 **Dirty Read**는 막지만, 같은 트랜잭션 안에서 다시 조회하면 다른 트랜잭션이 커밋한 변경 결과가 보일 수 있다.
- **Repeatable Read**
    - 같은 트랜잭션 안에서는 이전에 읽었던 데이터에 대해 **일관된 버전**을 보려고 한다.
    - 따라서 반복 조회 시 같은 row의 값이 바뀌는 **Non-repeatable Read**를 막는 데 초점을 둔다.
- **Serializable**
    - 트랜잭션 결과가 직렬 실행된 것처럼 보이도록 더 엄격하게 제어한다.
    - 이를 위해 MVCC만으로 부족한 경우 **Lock이나 충돌 제어를 더 강하게 사용**한다.

#### 유의 사항

H2도 격리 수준을 지원하지만, **MySQL**이나 **PostgreSQL**의 세부 **Lock** 동작을 그대로 재현하지는 않는다.

따라서 동시성 검증은 가능하면 실제 운영 DB와 같은 DBMS에서 수행하는 것이 안전하다.

### MySQL InnoDB

MySQL InnoDB는 **Undo Log 기반 MVCC**와 **Lock**을 조합해 트랜잭션 동시성을 제어한다.

레코드가 변경되면 기존 값은 **Undo Log**에 남고, 현재 레코드에는 최신 값이 저장된다.

트랜잭션은 **Read View**를 기준으로 현재 레코드가 자신에게 보이는 버전인지 판단한다.

만약 현재 레코드가 보이면 안 되는 버전이라면, Undo Log를 따라가며 자신이 볼 수 있는 과거 버전을 찾아 읽는다.

즉, **Read View**는 row version의 **가시성을 판단하는 스냅샷 정보**이고, **Undo Log**는 그 판단 결과에 따라 필요한 과거 버전을 복원하는 데 사용된다.

격리 수준별 동작은 다음과 같이 이해할 수 있다.

- **Read Committed**
    - SELECT를 실행할 때마다 새로운 **Read View**를 만든다.
    - 따라서 조회 시점에 커밋된 최신 데이터를 읽으며, 같은 트랜잭션 안에서도 다시 조회하면 다른 트랜잭션이 커밋한 변경 결과가 보일 수 있다.
- **Repeatable Read**
    - 첫 번째 consistent read 시점에 만든 **Read View**를 트랜잭션 동안 재사용한다.
    - 따라서 같은 트랜잭션 안에서는 반복 조회 시 같은 row의 값이 바뀌는 **Non-repeatable Read**를 막을 수 있다.
- **Serializable**
    - 일반적인 consistent read보다 더 강하게 제어한다.
    - 단순 조회도 잠금 읽기처럼 처리되어, 트랜잭션들이 직렬 실행된 것에 가까운 결과를 만들도록 제어한다.

### PostgreSQL

PostgreSQL은 **Tuple Version 기반 MVCC**와 **Lock / 충돌 감지**를 조합해 트랜잭션 동시성을 제어한다.

PostgreSQL은 **MySQL**처럼 **Undo Log**를 따라 과거 버전을 찾는 방식이 아니라, 테이블 내부에 row의 여러 버전, 즉 **tuple version**을 남겨둔다.

각 트랜잭션은 **Snapshot**을 기준으로 자신이 볼 수 있는 tuple version을 선택해 읽는다.

격리 수준별 동작은 다음과 같이 이해할 수 있다.

- **Read Committed**
    - SQL 문장을 실행할 때마다 새로운 **Snapshot**을 만든다.
    - 따라서 조회 시점에 커밋된 최신 데이터를 읽으며, 같은 트랜잭션 안에서도 다시 조회하면 다른 트랜잭션이 커밋한 변경 결과가 보일 수 있다.
- **Repeatable Read**
    - 트랜잭션 시작 시점의 **Snapshot**을 기준으로 계속 읽는다.
    - 따라서 같은 트랜잭션 안에서는 반복 조회 시 같은 row의 값이 바뀌는 **Non-repeatable Read**를 막을 수 있다.
- **Serializable**
    - PostgreSQL은 **SSI, Serializable Snapshot Isolation** 방식으로 구현한다.
    - 단순히 모든 읽기에 강한 Lock을 거는 방식이 아니라, 직렬 실행과 충돌할 수 있는 상황을 감지하면 특정 트랜잭션을 실패시킨다.
