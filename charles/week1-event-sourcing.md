# 이벤트 소싱이란?

### 상태가 아닌 이벤트(사건)를 저장하는 방식

- 이벤트 소싱이란 무엇일까? 어플리케이션의 **모든 상태 변화**를 **순서에 따라 이벤트로 보관하는 저장 패턴**이다.
- 현재 상태 값을 저장/수정/삭제해오던 전통적인 데이터 저장 방식과 다르게, **순차적으로 발생한 이벤트들을 모두 저장하는 방식이다.**

### 사건을 **저장함으로써** 얻을 수 있는 이점

상태 값 자체가 아닌 사건(이벤트)를 저장함으로 얻을 수 있는 이점은 무엇일까?

1. **과거의 이력 정보를 한번에 관리할 수 있다.**
    - 전통적인 데이터 저장 방식은 이전의 상태를 알 수 없다. 즉, 과거의 이력 정보가 필요한 경우, 별도의 테이블로 구성해서 관리해야 한다.
    - 비즈니스가 확장될 수록 이력 테이블은 더 짙고 복잡해진다. 이 경우 테이블을 찾고 기존 값을 업데이트 하는 방식은 매핑 과정에서 더 복잡한 로직이나 시간을 요구한다. 따라서 관계가 없는 심플한 이벤트만 저장하는 방식, 즉 오직 Append만 하는 방식으로 이벤트를 소싱한다.
2. **오로지 Event는 불변이기 때문에 오로지 Append 방식만 가능하다. 즉, Update와 Delete가 없기에 쓰기 성능이 올라간다.**
    - 각 이벤트는 불변하고, 오직 Append(추가)만 가능하다라는 특징을 가지며, 한번 추가된 이벤트는 수정되지 않는다. 이벤트는 각 식별자와 타입, 버전, 발생시간과 내용인 payload를 갖는다. 이 이벤트가 강력한 이유는 관계가 없는 매우 심플한 구성이기 때문이다.

### 이벤트 소싱 도입 예시

- 도서 구매 어플리케이션을 예로 들어보자.
- 사용자 A가 1번 책과 2번 책을 순차적으로 장바구니에 추가하고, 2번 책을 삭제했다고 가정하자.
<img width="4095" height="2061" alt="image" src="https://github.com/user-attachments/assets/f30d8376-3490-42a9-8c57-ce89610dc051" />


- 기존의 결과 값만 저장하던 방식은 장바구니에 1번 책만 담긴 상태로 DB에 저장되어 있겠지만(사진의 current status), 이벤트 소싱의 경우 1번 추가, 2번 추가, 2번 삭제라는 3가지의 이벤트로 DB에 저장되어 있다. 즉, 이벤트 소싱에서는 Delete와 Update의 개념이 없다. 오로지 이벤트가 생성되는 것만 존재한다.

### 현재 상태는 어떻게 불러오는가

- 이벤트 소싱에서는 전통적인 방식대로 현재 상태 값을 그대로 저장하는 것이 아닌, 이벤트를 저장한다. 그렇기에 현재 상태 값이 필요하다면 DB에 저장된 이벤트들을 기반으로 어플리케이션 메모리에서 재구성해야 한다.
- 이를 Event Replay라고 한다. 즉, 이벤트를 조회하고 객체의 현상태를 복원한다. 최신 상태를 알기 위해서는 매번 replay를 해야한다. 매번 재구성해야 한다고?

### 스냅 샷이 필요한 이유

- 메모리 재구성의 치명적 단점은 무엇일까? 사용자가 같은 책을 10000000번 추가하고 삭제하기를 반복했다면 어떤 일이 벌어질까? 정작 장바구니의 현재 상태값은 비어있는데, 이를 재구성하기 위해 이벤트를 10000000번 재생해야 한다. 결국 현재 객체 상태를 불러오기까지의 시간에 트레이드 오프가 발생한다.
- 이를 보완하기 위해 스냅샷 개념을 도입한다. 스냅샷이란 특정 시점에 Replay된 도메인 객체 정보를 의미한다. 즉, 1000번 단위로 스냅샷을 각각 저장해 둔다면, 1001번째 상태 값이 필요한 경우, 1번 스냅샷을 기준으로 1번만 재생하면 된다.
- 그렇다면 스냅샷을 언제 저장할까? 이벤트 저장 시점에 생성한다. 하지만 매 이벤트 생성시점이 아닌, 서비스 정책에 맞게 시간 혹은 이벤트 발생 개수를 기준으로 스냅샷을 저장한다.
- Snapshot은 일반적으로 EventStore와 분리된 저장소에 저장해야 하는데 성능상 주로 in memory 저장소를 사용한다. Snapshot은 일반적으로 캐시의 성향이 짙다.

### 스냅 샷을 인메모리에 저장하는 이유

- 아래 상세 구현을 보면 알다시피, Snapshot은 같은 aggregate를 여러 번 로딩할 때 같은 데이터가 계속 읽힌다. Event는 매 요청마다 다른 이벤트 범위를 읽는 것과 달리, Snapshot은 반복적으로 같은 데이터가 접근되기 때문에 성능상 인메모리가 유리하다.

자 여기까지 정리해보자. 이벤트 소싱은 결과 값이 아닌 모든 상태 변화의 이벤트를 저장하는 방식이다. 그리고 이는 수정과 삭제가 아닌 오로지 추가의 방식(append)으로만 데이터가 관리된다. 그렇기에 현재의 도메인 상태를 불러오기 위해서는 이벤트를 불러와 replay 해야 하는데 재생 처리 성능을 위해 스냅샷을 일반적으로 도입한다. 

이벤트 소싱의 쓰기 시점을 생각해보자. 만약 장바구니라는 Aggregate에 대해 특정 Command(명령, 즉 상태 변경 요청) 요청이 들어왔다고 해보자. 

# CQRS란?

이벤트 소싱 방식은 CQRS와 궁합이 좋다고 한다. CQRS가 무엇인지 먼저 짚고 넘어가자.

### 이벤트 소싱의 단점

- 이벤트 소싱의 단점은 메모리 위에서 재생하기 위한 처리 시간이 병목으로 제한된다는 점에 있다. 이때 읽기 작업이 들어온다면, 현재 상태값을 알기 위해 모든 객체에 대해 재구성 작업을 진행해줘야 할 것이다. 이에 따라 읽기 모델을 별도로 분리하는 CQRS 패턴을 같이 사용해줘야 좋다.

### CQS와 CQRS란?

- CQS와 CQRS는 모두 command와 query의 책임을 분리하는 것을 의미한다. 다만 CQS는 객체/메서드와 같은 코드단위의 분리를 의미하고, CQRS는 아키텍처 패턴이나 시스템 레벨의 더 넓은 범위에서의 분리를 의미하는 용어이다.
- 코드 단위에서의 Command와 Query 분리(CQS)를 먼저 이해하고 넘어가자.
- 아래 예시 `isSatisfied()` 메서드를 살펴보자.

```java
private final int currentSize; 

public int isSatisfied(int threshold) {
    if (currentRate < threshold) {
        resize()   // 상태 변경 (Command)
        return false;      // 값 반환 (Query)
    }
    return true;
}

private void resize(){
		this.currentSize += 5;
}
```

해당 `isSatisfied()` 메서드는 인자값 `threshold`을 기준으로 현재 상태인 `currentSize`가 만족하는지 판단해주는 메서드이다. 이때, `isSatisfied()` 는 내부적으로 만족하지 않는 경우 현재 상태인 `currentSize` 를 재조정해주기 위해 `resize()` 메서드를 실행하고 있다. 이 경우 외부에서 isSatisfied를 호출하는 입장에서 호출때 마다 다른 결과 값이 나올 수 있다.

```java
block.isSatisfied(10) // false 반환
block.isSatisfied(10) // false 반환
...
block.isSatisfied(10) // true 반환
```

여기서 상태 변경인 명령과 조회 개념인 쿼리가 한 메서드에서 혼용되었기 때문에, 외부 호출부의 입장에서는 실행 결과를 예측하기 어려워질 수 있다. 이에 따라 쿼리와 커멘드를 메서드 단위에서 분리해야하는 것을 CQS라고 한다.

CQS를 사용하면 무엇이 좋을까? 우선 반환 값을 돌려주는 메서드는 쿼리이므로 Side Effect에 대한 부담이 없음을 개발자가 인지할 수 있다. 반면, 반환 값을 가지지 않는 void 메서드는 모두 명령으로 해석하여 내부 상태가 변경된다는, 즉 side effect가 발생한다는 점을 인지하고 주의할 수 있다.

이와 같이 명령과 쿼리를 시스템 단위에서 분리한 것을 CQRS라고 한다. 즉 DB 시스템 관점에서 바라본다면, Query는 DB조회, Command는 DB 수정/추가/삭제를 의미한다. 이에 따라 쓰기 서버와 읽기 서버를 분리하자라는 것이 CQRS 아키텍처를 따라가자라는 의미가 될 수 있다.

### 왜 CQRS가 이벤트 소싱과 궁합이 좋을까?

이벤트 소싱은 일단 이벤트가 저장만 된다고 했다. 결국 append(추가)만 하기 때문에 기존의 Update가 아닌 Insert만 존재하기 때문에 쓰기 경합이 줄어들고 성능이 더 좋아질 수 있다. 하지만 조회 성능을 전통적인 방식에 비하면 월등히 떨어진다. 읽기 서버를 따로 분리한다면 projection이나 read model을 만들어 조회 최적화도 가능한데, 이를 살펴보자.

100만개의 계좌를 대상으로 최근 일주일 안에 거래 정지된 계좌를 찾아보자. 그럼 총 100만개의 객체를 다 재생하고 계좌를 찾아야 할 것이다. 객체의 재생 속도는 이벤트의 개수에 따른 차이가 있겠지만, 스냅샷을 적용했다 하더라도 전통적인 저장 방식보다 훨씬 더 많은 시간이 소요된다. 

이때 CQRS 패턴을 도입한다면, 쓰기 모델과 읽기 모델을 분리해서 사용한다면 조회 성능을 해결 할 수 있다. 조회(Query) 전용 모델을 별도로 구현하면 된다. 이때 Projection을 통해 우리에게 익숙한 형태의 조회 모델을 구성한다.

## 3.1 Command(명령) 시퀀스

<img width="10703" height="9785" alt="image 2" src="https://github.com/user-attachments/assets/dacafd77-9d60-40fa-8337-7f236d66c8ab" />


**Command는 Aggregate를 이벤트로 복원한 뒤, 새로운 이벤트를 만들어 append하는 과정이다.**

1. command 요청이 들어온다. 이때, command 객체를 만들고 이를 이벤트 핸들러로 보낸다. 여기서 command 객체는 우리가 일반적으로 입력받는 DTO로 해석하면 된다.
2. 대상 Aggregate를 재생(replay)한다. 여기서 Aggregate는 비즈니스 로직을 포함하는 도메인 객체를 의미한다. 즉 Aggregate 내부에서 비즈니스 로직 실행 이후의 이벤트를 발행하는 책임을 갖는다. 재생시에는 snapshot을 우선적으로 조회하고 이를 기반으로 재생한다.  
3. 재생된 Aggregate 내부에서 비즈니스 로직이 실행된다. 이 실행 결과는 상태값 수정이 아닌 이벤트의 형식으로 메모리에 저장된다.
4. 이후, EventStore(영구 저장소)에 해당 event가 저장되며, 스냅샷 정책에 만족하는 경우 Snapshot은 인메모리에 저장된다. snapshot은 캐시 개념으로 이해하면 편하다.
5. 마지막으로 snapshot 비동기로 Event가 발행되고 projection을 통해 읽기 모델이 업데이트 된다.

중요한 점

- 도메인 로직 실행시 상태를 바로 바꾸지 않는다.
- 이벤트는 커밋 단위로 저장된다.
- 여기서 핵심은 CQRS 패턴을 도입해, 매번 재구성해야하는 읽기 단점의 성능을 보안했다는 점이다.

### 3.2 Query(조회)시퀀스

조회 메서드는 읽기 모델로 요청이 들어간다. 즉, 명령 요청시 projection을 통해 만들어진 읽기 모델을 읽기 때문에 별도의 replay가 진행되지 않는다. 즉, 읽기 성능을 포기하지 않을 수 있다.

**해당 매커니즘으로 이벤트 소싱을 적용하는 경우 얻을 수 있는 장점**

- 동시성 문제를 해결할 수 있다. 각 어그리게이트 재구성시점에 버전 정보를 저장한다. 이 버전 정보를 기반으로 저장 시점에 버전 체크를 진행하기에 동시성 문제를 해결 가능하다. (lost update 방지)
- 감사로그와 같이 이력이 남아야하는 경우 관리하기 편하다. 앞서 살펴봤듯이, 이력 테이블을 별도로 관리하지 않기에 이벤트만 재생해서 관리하기 용이하다.
- 조회와 읽기 모델의 분리로 읽기 성능도 올라간다.
- 디버깅 시점에 replay로 버그를 직접 재현 가능하다.
