## 2. Lettuce vs Jedis

Spring Boot에서 Redis에 연결하는 클라이언트 라이브러리는 두 가지가 있다.

### 클라이언트 라이브러리란?

Redis 서버는 별도 프로세스로 실행된다. Java 애플리케이션이 Redis에 명령을 보내려면 중간에 "통역사" 역할을 하는 라이브러리가 필요하다. 이것이 클라이언트 라이브러리다.

### 비교

| 항목 | Lettuce | Jedis |
| --- | --- | --- |
| 처리 방식 | 비동기 / Non-blocking | 동기 / Blocking |
| 스레드 안전 | 하나의 커넥션을 여러 스레드가 공유 가능 | 스레드마다 별도 커넥션 필요 |
| 연결 풀 | 불필요 (기본적으로 연결 공유) | 필요 (커넥션 풀 설정 권장) |
| Spring Boot 기본값 | **Lettuce** | 별도 의존성 추가 필요 |

### Non-blocking이란?

**Blocking 방식 (Jedis)**

Redis에 명령을 보내고 응답이 올 때까지 해당 스레드가 멈춰서 기다린다. 그 사이에 다른 요청이 오면 새 스레드가 필요하다.

```
Thread-1: [Redis 명령 전송] --- 기다리는 중 --- [응답 수신] [다음 작업]
Thread-2:                        [Redis 명령 전송] --- 기다리는 중 --- [응답 수신]
```

**Non-blocking 방식 (Lettuce)**

명령을 보내고 기다리지 않는다. 응답이 오면 그때 처리한다. 하나의 커넥션으로 여러 요청을 처리할 수 있다.

```
Thread-1: [명령 전송] [다른 작업] [응답 수신 후 처리]
Thread-1: [명령 전송] [다른 작업] [응답 수신 후 처리]
```

`spring-boot-starter-data-redis`를 추가하면 Lettuce가 기본으로 사용된다. Jedis는 멀티스레드 환경에서 커넥션 풀을 직접 관리해야 하는 번거로움이 있어, 특별한 이유가 없으면 Lettuce를 사용하는 것이 일반적이다.

## 3. StringRedisTemplate

### RedisTemplate이란?

Spring Data Redis에서는 Redis에 접근하기 위해 `RedisTemplate` 계열의 클래스를 제공한다. Redis 서버와 주고받는 데이터를 자바 객체로 변환해주는 역할을 한다.

### StringRedisTemplate

`StringRedisTemplate`은 **키(Key)와 값(Value)을 모두 문자열(String)로 다루는 RedisTemplate의 특수화 버전**이다.

```java
public class StringRedisTemplate extends RedisTemplate<String, String>
```

내부적으로는 다음과 같은 직렬화 방식을 사용한다.

- Key → `StringRedisSerializer`
- Value → `StringRedisSerializer`

즉, Redis에 저장되는 데이터가 사람이 읽을 수 있는 문자열 형태로 저장된다.

### 직렬화(Serialization)란?

자바 객체는 메모리 안에 복잡한 구조로 존재한다. Redis에 저장하려면 이를 바이트(byte) 형태로 변환해야 한다. 이 과정을 직렬화라고 한다.

```
자바 객체 → (직렬화) → 바이트/문자열 → Redis 저장
Redis 저장 → (역직렬화) → 자바 객체
```

`StringRedisSerializer`는 문자열을 그대로 UTF-8 바이트로 변환하므로, Redis에 저장된 값을 `redis-cli`로 직접 봤을 때 사람이 읽을 수 있는 형태다.

### 사용 예시

```java
stringRedisTemplate.opsForValue().set("online:user:1", "1");
```

Redis에는 실제로 다음과 같이 저장된다.

```
online:user:1 = "1"
```

### 왜 StringRedisTemplate을 선택했나

이번 온라인 상태 관리 기능에서 저장하는 데이터는 모두 문자열이었다.

- 온라인 플래그 → `"1"`
- 사용자 ID 기반 키 → `"online:user:1"`
- WebSocket 세션 ID → `"sessionId-abc"`

복잡한 객체를 직렬화/역직렬화할 필요가 없으므로, 별도 설정 없이 바로 쓸 수 있는 `StringRedisTemplate`이 가장 단순하고 직관적인 선택이었다.

> 만약 User 객체 같은 복잡한 데이터를 저장해야 한다면 `RedisTemplate<String, Object>`에 `Jackson2JsonRedisSerializer`를 등록하는 방식을 사용한다.

## 4. 사용한 Redis 메서드

### opsFor~ 패턴이란?

Redis는 자료구조마다 사용할 수 있는 명령어가 다르다.

- String: `SET`, `GET`, `SETEX`
- Set: `SADD`, `SREM`, `SCARD`
- Hash: `HSET`, `HGET`
- List: `LPUSH`, `RPOP`

Spring은 `opsFor~` 메서드로 자료구조를 먼저 선택한 뒤 해당 명령어를 사용하도록 설계했다.

```java
// opsFor~ 없이 설계했다면 (혼란스러운 예시)
redisTemplate.add("key", "value");      // Set용인지 List용인지 모름
redisTemplate.set("key", "value");      // String용인지 Hash용인지 모름

// opsFor~ 사용 (자료구조가 명확히 보임)
redisTemplate.opsForSet().add(...);     // Set에 추가
redisTemplate.opsForHash().put(...);    // Hash에 추가
redisTemplate.opsForValue().set(...);   // String에 저장
```

| 메서드 | 대상 자료구조 | Redis 명령어 |
| --- | --- | --- |
| `opsForValue()` | String (단순 키-값) | `SET`, `GET`, `SETEX` |
| `opsForSet()` | Set (중복 없는 집합) | `SADD`, `SREM`, `SCARD` |
| `opsForList()` | List (순서 있는 목록) | `LPUSH`, `RPOP` |
| `opsForHash()` | Hash (필드-값 맵) | `HSET`, `HGET` |
| `opsForZSet()` | Sorted Set (점수 기반 정렬 집합) | `ZADD`, `ZRANGE` |

### opsForValue — 단순 문자열 키-값

온라인 플래그 저장에 사용한다. `online:user:1 = "1"` 형태로 저장되며, TTL이 지나면 자동 삭제된다.

| 메서드 | 설명 |
| --- | --- |
| `opsForValue().set(key, value, ttl)` | 키에 값을 저장하고 TTL 설정 |

```java
stringRedisTemplate.opsForValue().set(
    "online:user:1",
    "1",
    Duration.ofSeconds(90)
);
```

### opsForSet — Set 자료구조

세션 목록 관리에 사용한다. Java의 `HashSet`과 동일하게 **중복 없는 문자열 모음**을 저장한다.

```
online:user:1:sessions = {
    "sessionA",
    "sessionB",
    "sessionC"
}
```

세션 목록에 Set을 선택한 이유는 중복이 없어야 하고 순서도 필요 없기 때문이다. 같은 세션이 두 번 추가돼도 하나만 저장된다.

| 메서드 | 설명 | Redis 명령어 |
| --- | --- | --- |
| `opsForSet().add(key, value)` | Set에 값 추가 | `SADD` |
| `opsForSet().remove(key, value)` | Set에서 값 제거 | `SREM` |
| `opsForSet().size(key)` | Set의 원소 개수 반환 | `SCARD` |

```java
// 세션 추가 (다기기 접속 시 세션 ID 등록)
stringRedisTemplate.opsForSet().add("online:user:1:sessions", "sessionId-abc");

// 세션 제거 (특정 기기 종료 시)
stringRedisTemplate.opsForSet().remove("online:user:1:sessions", "sessionId-abc");

// 남은 세션 수 확인 (0이면 모든 기기 오프라인)
stringRedisTemplate.opsForSet().size("online:user:1:sessions");
```

### 키 관리 메서드

| 메서드 | 설명 |
| --- | --- |
| `expire(key, duration)` | 기존 키에 TTL 설정 또는 갱신 |
| `delete(key)` | 키 삭제 |
| `hasKey(key)` | 키 존재 여부 반환 (Boolean) |

`expire()`는 `opsForSet().add()`로 세션을 추가한 뒤 TTL을 별도로 갱신할 때 쓴다. `add()` 자체는 TTL을 설정하지 않기 때문에 따로 호출해야 한다.

```java
// Set 키에 TTL 갱신
stringRedisTemplate.expire("online:user:1:sessions", Duration.ofSeconds(90));

// 온라인 키 삭제 (오프라인 처리)
stringRedisTemplate.delete("online:user:1");

// 온라인 여부 확인 (키가 존재하면 온라인)
stringRedisTemplate.hasKey("online:user:1"); // true면 온라인
```

## 5. 배경 및 설계 목표

Spring Boot + STOMP WebSocket 기반 채팅 서비스에서 채팅방 멤버의 온라인 여부를 실시간으로 표시하는 기능을 구현했다.

이 기능에서 **"온라인"은 로그인 여부가 아니라, 채팅방에 실제로 WebSocket이 연결되어 메시지를 주고받을 수 있는 상태**를 의미한다.

React 클라이언트가 채팅방 페이지에 진입할 때 CONNECT, 이탈할 때 DISCONNECT가 발생하므로 온라인 상태는 "로그인 중"이 아닌 "채팅방 접속 중"에 대응한다.

### STOMP란?

STOMP(Simple Text Oriented Messaging Protocol)는 WebSocket 위에서 동작하는 메시지 프로토콜이다. 날것의 WebSocket은 텍스트/바이너리 프레임만 있어서 "이 메시지가 어디로 가야 하는가"를 직접 구현해야 한다. STOMP는 `CONNECT`, `SEND`, `SUBSCRIBE`, `DISCONNECT` 같은 명령어를 정의해서 메시지 라우팅을 편리하게 해준다.

```
클라이언트 → CONNECT 프레임  → 서버 (연결 수립)
클라이언트 → SEND 프레임     → 서버 (메시지 전송)
클라이언트 → DISCONNECT 프레임 → 서버 (연결 해제)
```

## 6. DB 대신 Redis를 선택한 이유

### DB로 구현했다면

처음에 `user_online_status` 테이블에 `is_online` boolean과 `last_seen_at` timestamp를 두는 방식을 고려했다.

```sql
CREATE TABLE user_online_status (
    user_id      BIGINT PRIMARY KEY,
    is_online    BOOLEAN,
    last_seen_at DATETIME
);
```

STOMP CONNECT 시 UPDATE, DISCONNECT 시 UPDATE. 구현 자체는 단순하다.

### 문제 1 — 데이터 성격이 맞지 않는다

온라인 상태는 앱을 껐다 켜면 초기화되는 **휘발성 데이터**다. 서버가 재시작되면 모든 WebSocket 세션이 끊기므로 기존에 `is_online = true`였던 레코드는 전부 stale(오래된) 상태가 된다.

DB는 **영속성(지속적으로 보존해야 하는)** 데이터를 위한 저장소다. 온라인 상태처럼 언제든 초기화될 수 있는 데이터를 DB에 저장하는 것은 저장소의 목적과 맞지 않는다.

### 문제 2 — 비정상 종료 처리를 직접 구현해야 한다

STOMP DISCONNECT 프레임은 **정상 종료에서만** 서버에 도달한다.

- 브라우저 탭 강제 종료
- 네트워크 끊김
- 앱 강제 종료

위 상황에서는 DISCONNECT가 오지 않는다. DB 방식이라면 `is_online = true`가 영구히 남아 있기 때문에, 주기적으로 오래된 레코드를 정리하는 배치 잡을 별도로 만들어야 한다.

```java
// DB 방식이라면 필요한 배치 잡
@Scheduled(fixedDelay = 60000)
public void cleanUpOfflineUsers() {
    // 마지막 접속 시간이 2분 이전인 유저를 오프라인으로 일괄 업데이트
    userRepository.markOfflineIfLastSeenBefore(LocalDateTime.now().minusMinutes(2));
}
```

Redis TTL을 활용하면 별도의 정리 배치 없이 일정 시간 내 자동으로 오프라인 처리된다.

### 문제 3 — CONNECT / DISCONNECT마다 DB 쓰기가 발생한다

채팅 앱에서 유저가 채팅방에 들어가고 나오는 행위는 매우 빈번하다. 그때마다 DB에 UPDATE를 치면 트래픽이 몰릴 때 DB에 상당한 부하가 생긴다.

| | DB | Redis |
| --- | --- | --- |
| 속도 | 디스크 I/O 발생 | 메모리에서 처리 |
| 만료 처리 | 배치 잡 별도 구현 | TTL로 자동 처리 |
| 데이터 성격 | 영속성 데이터 | 휘발성 데이터 |
| 부하 | CONNECT/DISCONNECT마다 DB 쓰기 | 메모리 읽기/쓰기로 부하 적음 |

온라인 상태처럼 **자주 바뀌고, 만료가 필요하고, 영속성이 필요 없는 데이터**는 Redis가 DB보다 훨씬 적합하다.

## 7. Redis 설계

### Redis 키 구조

유저 한 명당 두 가지 키를 관리한다.

```
online:user:{userId}          → "1"              (온라인 플래그, TTL 90초)
online:user:{userId}:sessions → Set<sessionId>   (연결된 WebSocket 세션 목록)
```

### 온라인 플래그 하나만 쓰면 안 되는 이유

가장 단순한 구현은 `online:user:{userId} = "1"` 하나만 두는 것이다. 그런데 이 방식은 **다중 기기 접속**에서 버그가 생긴다.

```
폰 접속     → online = "1"
노트북 접속  → online = "1"
폰 종료     → online 키 삭제  ← 노트북이 아직 접속 중인데 오프라인 처리됨!
```

이를 해결하기 위해 각 WebSocket 세션을 Set에 따로 관리한다.

```
폰 접속     → sessions = {A},    online = "1"
노트북 접속  → sessions = {A, B}, online = "1"
폰 종료     → sessions = {B},    online = "1"  ← B가 남아있으므로 유지
노트북 종료  → sessions = {},    online 키 삭제 → 오프라인
```

`markOffline()`은 세션을 제거한 후 **Set이 비어있을 때만** online 키를 삭제한다.

### TTL(Time To Live) 90초의 역할

TTL은 Redis 키의 **자동 만료 시간**이다. 지정한 시간이 지나면 Redis가 자동으로 해당 키를 삭제한다.

비정상 종료 시 DISCONNECT 프레임이 오지 않아 `markOffline()`이 호출되지 않는다. TTL이 안전망 역할을 해서 90초가 지나면 키가 자동 만료되어 오프라인 처리된다.

**90초를 선택한 이유**

- TTL이 너무 짧으면 (예: 10초): 모바일 환경에서 일시적인 네트워크 단절 후 재연결되는 동안 오프라인으로 표시될 수 있다. 일반적인 WebSocket 재연결 타임아웃은 30~60초 수준이다.
- TTL이 너무 길면 (예: 10분): 브라우저 강제 종료 후 10분간 온라인으로 보인다. 메시지를 받을 수 없는 상태인데도 온라인으로 표시되어 혼란을 준다.

WebSocket 재연결 시간보다 충분히 길고, 비정상 종료 후 오프라인 처리가 너무 늦지 않도록 경험적으로 90초를 선택했다.

## 8. 구현

### 의존성 추가

```gradle
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
```

### ChatPresenceService

온라인 상태를 Redis에 읽고 쓰는 핵심 서비스다.

```java
@Service
@RequiredArgsConstructor
public class ChatPresenceService {

    private static final Duration ONLINE_TTL = Duration.ofSeconds(90);
    private static final String ONLINE_KEY_PREFIX = "online:user:";
    private static final String SESSION_KEY_SUFFIX = ":sessions";

    private final StringRedisTemplate stringRedisTemplate;

    public void markOnline(Long userId, String sessionId) {
        String onlineKey = onlineKey(userId);
        String sessionKey = sessionKey(userId);

        // 세션 Set에 현재 접속 세션 ID 추가 (다중 기기 접속 대응)
        stringRedisTemplate.opsForSet().add(sessionKey, sessionId);
        // 세션 Set의 TTL 갱신 (접속할 때마다 만료 시간 초기화)
        stringRedisTemplate.expire(sessionKey, ONLINE_TTL);
        // 온라인 플래그 저장 (키 존재 여부로 온라인 상태 판단)
        stringRedisTemplate.opsForValue().set(onlineKey, "1", ONLINE_TTL);
    }

    public void markOffline(Long userId, String sessionId) {
        String onlineKey = onlineKey(userId);
        String sessionKey = sessionKey(userId);

        // 종료된 세션만 Set에서 제거 (다른 기기 세션은 유지)
        stringRedisTemplate.opsForSet().remove(sessionKey, sessionId);

        Long sessionCount = stringRedisTemplate.opsForSet().size(sessionKey);
        // 남은 세션이 없으면 모든 기기가 오프라인이므로 키 전체 삭제
        if (sessionCount == null || sessionCount <= 0) {
            stringRedisTemplate.delete(sessionKey);
            stringRedisTemplate.delete(onlineKey);
        }
    }

    public Map<Long, Boolean> getOnlineStates(List<Long> userIds) {
        // 유저 ID 목록을 받아 각각의 온라인 플래그 키 존재 여부를 Map으로 반환
        return userIds.stream()
                .filter(Objects::nonNull)
                .collect(Collectors.toMap(
                        userId -> userId,
                        userId -> Boolean.TRUE.equals(stringRedisTemplate.hasKey(onlineKey(userId)))
                ));
    }

    private String onlineKey(Long userId) {
        return ONLINE_KEY_PREFIX + userId;
    }

    private String sessionKey(Long userId) {
        return ONLINE_KEY_PREFIX + userId + SESSION_KEY_SUFFIX;
    }
}
```

### StompHandler — CONNECT / DISCONNECT 감지

STOMP 메시지가 처리되기 전에 실행되는 `ChannelInterceptor`를 구현해 연결/해제를 감지한다.

`ChannelInterceptor`는 Spring Messaging에서 메시지 채널을 통과하는 메시지를 가로채는 인터페이스다. `preSend()`는 메시지가 채널로 전달되기 **전에** 실행된다.

```java
@Component
@RequiredArgsConstructor
public class StompHandler implements ChannelInterceptor {

    private final JwtTokenProvider jwtTokenProvider;
    private final ChatPresenceService chatPresenceService;

    @Override
    public Message<?> preSend(Message<?> message, MessageChannel channel) {
        StompHeaderAccessor accessor = MessageHeaderAccessor.getAccessor(message, StompHeaderAccessor.class);
        if (accessor == null || accessor.getCommand() == null) return message;

        // CONNECT 프레임: JWT 파싱 후 온라인 상태로 전환
        if (StompCommand.CONNECT.equals(accessor.getCommand())) {
            Authentication auth = resolveAuthentication(accessor);
            if (auth != null) {
                accessor.setUser(auth);
                chatPresenceService.markOnline(Long.parseLong(auth.getName()), accessor.getSessionId());
            }
        }

        // DISCONNECT 프레임: 해당 세션 오프라인 처리
        if (StompCommand.DISCONNECT.equals(accessor.getCommand())) {
            Authentication auth = (Authentication) accessor.getUser();
            if (auth != null) {
                chatPresenceService.markOffline(Long.parseLong(auth.getName()), accessor.getSessionId());
            }
        }

        return message;
    }
}
```

### ChatRoomService — 채팅방 조회 시 온라인 상태 반영

채팅방 멤버 목록을 조회할 때 `getOnlineStates()`로 Redis를 조회해서 DTO에 담는다.

```java
public ChatRoomResDto getChatRoom(Long roomId, Long userId) {
    ChatRoom chatRoom = chatRoomRepository.findChatRoomByIdAndUserId(roomId, userId)
            .orElseThrow(() -> new ChatException(ChatErrorCode.CHAT_ROOM_ACCESS_DENIED));

    // 채팅방 멤버 전체 ID 추출
    List<Long> memberIds = chatRoom.getChatRoomUsers().stream()
            .map(cru -> cru.getUser().getId())
            .toList();

    // Redis에서 멤버 전체의 온라인 상태 조회
    Map<Long, Boolean> onlineStates = chatPresenceService.getOnlineStates(memberIds);

    // 각 멤버 DTO에 온라인 상태를 함께 담아 응답 구성
    List<ChatRoomUserResDto> users = chatRoom.getChatRoomUsers().stream()
            .map(cru -> ChatRoomUserResDto.from(
                    cru,
                    onlineStates.getOrDefault(cru.getUser().getId(), false)
            ))
            .toList();
```

## 9. 전체 흐름

### 정상 흐름

```
1. 채팅방 진입
   클라이언트 → WebSocket CONNECT
   StompHandler → JWT 파싱 → markOnline(userId, sessionId)
   Redis: online:user:1 = "1" (TTL 90s)
          online:user:1:sessions = {"sessionId-abc"}

2. 채팅방 조회 API 호출
   GET /api/chat/rooms/{roomId}
   ChatRoomService → getOnlineStates([1L, 2L])
   Redis에서 키 존재 여부 확인 → online 필드 true/false로 응답

3. 채팅방 이탈
   클라이언트 → WebSocket DISCONNECT
   StompHandler → markOffline(userId, sessionId)
   sessions Set 비어있음 → online 키 삭제 → 오프라인
```

### 비정상 종료 (강제 종료 / 네트워크 끊김)

```
DISCONNECT 프레임 서버에 도달하지 않음
→ markOffline() 호출 안 됨
→ TTL 90초 경과 후 Redis 키 자동 만료
→ 오프라인 처리 (최대 90초 지연)
```

### 다중 기기 접속

```
폰 접속     → sessions = {A},    online = "1"
노트북 접속  → sessions = {A, B}, online = "1"
폰 종료     → sessions = {B},    online = "1"  (B가 남아있으므로 유지)
노트북 종료  → sessions = {},    online 키 삭제 → 오프라인
```
