## 1. Redis란?

Redis는 `Remote Dictionary Server`의 약자이다.

- 원격(Remote)에 있는 Dictionary(Map 형태 자료구조) 서버
- 고성능 Key-Value 기반 인메모리 데이터 저장소
- NoSQL 데이터베이스 중 하나
- 메모리(RAM)에 데이터를 저장하여 매우 빠른 속도를 제공

Redis는 단순 문자열 저장뿐 아니라 다양한 자료구조를 제공하며 캐시, 세션, 실시간 시스템 등에 많이 사용된다.

대표 사용 사례

- 캐시(Cache)
- 세션(Session)
- 실시간 랭킹
- 채팅 시스템
- 대기열 시스템
- 분산 락
- API Rate Limiting

## 2. Redis의 특징

### (1) 인메모리 기반 데이터 저장소

Redis는 데이터를 디스크가 아니라 메모리에 저장한다.

그 결과

- 읽기/쓰기 속도가 매우 빠름
- 디스크 기반 데이터베이스보다 높은 성능 제공
- 실시간 처리에 적합

하지만 메모리는 휘발성이기 때문에 서버가 종료되면 데이터가 사라질 수 있다.

따라서 데이터 영속성이 필요하다면 별도의 Persistence 설정이 필요하다.

또한 Redis는 메모리 기반 시스템이기 때문에 디스크 IO가 거의 발생하지 않는다.

Redis는 메모리 기반 구조 덕분에 빠른 응답 속도를 유지할 수 있다.

### (2) 다양한 자료구조 제공

Redis는 단순 Key-Value 저장소를 넘어 다양한 자료구조를 지원한다.

이 자료구조들은 프로그래밍 언어의 기본 자료구조와 유사하기 때문에 애플리케이션 개발을 단순하게 만든다.

또한 관계형 데이터베이스와 프로그래밍 언어 사이에서 발생하는 임피던스 불일치를 줄일 수 있다.

- 임피던스 불일치(Impedance Mismatch)
    - 관계형 데이터베이스의 테이블 구조와 프로그래밍 언어의 객체 구조 차이로 인해 발생하는 문제
- Redis 자료구조

| 자료구조 | 설명 | 사용 예시 |
| --- | --- | --- |
| String | 가장 기본적인 데이터 타입 | 일반 데이터 저장 |
| List | 순서가 있는 문자열 리스트 | 대기열, 채팅 |
| Set | 중복을 허용하지 않는 집합 | 방문자 관리 |
| Sorted Set(ZSet) | 점수(score) 기준 정렬 | 랭킹 시스템 |
| Hash | Field-Value 구조 | 객체 저장 |
| Bitmap | 비트 단위 연산 | 출석 체크 |
| HyperLogLog | 대량 데이터 집계 | 방문자 수 추정 |

### (3) Persistence(데이터 영속성)

Redis는 기본적으로 인메모리 데이터베이스이다.

하지만 데이터를 디스크에 저장하여 영구적으로 유지할 수 있는 기능을 제공한다.

대표적인 방식

- RDB
- AOF

실무에서는 두 방식을 함께 사용하는 경우가 많다.

#### a. RDB (Redis Database Backup)

특정 시점의 메모리 데이터를 스냅샷 형태로 저장하는 방식이다.

저장 결과

- dump.rdb 파일 생성

RDB 동작 방식

1. Redis가 `BGSAVE` 명령 실행
2. `fork()` 호출하여 자식 프로세스 생성
3. 자식 프로세스가 메모리 데이터를 dump.rdb 파일로 저장
4. 부모 프로세스는 계속 클라이언트 요청 처리
5. 저장 완료 후 자식 프로세스 종료

장점

- 성능이 좋음
- 복구 속도가 빠름
- 파일 크기가 비교적 작음

단점

- 마지막 스냅샷 이후 데이터는 유실될 수 있음

#### b. AOF (Append Only File)

모든 변경 명령을 로그 파일에 기록하는 방식이다.

예시

- SET user hong
- INCR count
- LPUSH queue a

같은 명령어들이 appendonly.aof 파일에 저장된다.

AOF 동작 방식

1. 모든 변경 명령을 AOF 로그 파일에 기록
2. 로그 파일 크기가 커지면 Rewrite 수행
3. Rewrite 과정에서도 `fork()`를 사용하여 백그라운드 작업 수행
4. Redis 재시작 시 AOF 파일 명령을 다시 실행하여 데이터 복구

장점

- 데이터 안정성이 높음
- 데이터 유실 가능성이 적음

단점

- 로그 파일 크기가 커질 수 있음
- 성능 오버헤드 존재

RDB와 AOF 비교

| 구분 | RDB | AOF |
| --- | --- | --- |
| 저장 방식 | 스냅샷 | 명령 로그 |
| 성능 | 좋음 | 상대적으로 느림 |
| 데이터 안정성 | 일부 유실 가능 | 안정성 높음 |
| 복구 속도 | 빠름 | 상대적으로 느림 |
| 파일 크기 | 비교적 작음 | 커질 수 있음 |

### (4) 싱글 스레드 이벤트 루프 모델

Redis의 가장 대표적인 특징 중 하나이다.

Redis는 기본적으로

- 메인 스레드 1개
- 보조 스레드 3개

구조로 동작한다.

하지만 핵심 데이터 처리 작업은 메인 스레드 하나에서 수행된다.

즉, 전체적으로는 멀티 스레드 요소가 존재하지만 실제 Redis 명령 실행은 싱글 스레드 기반이다.

#### a. 메인 스레드가 하는 일

메인 스레드는 Redis의 핵심 데이터 처리를 담당한다.

대표적인 작업

- `GET` : Key에 저장된 값을 조회한다.
- `SET` : Key에 값을 저장하거나 수정한다.
- `HSET` : Hash 자료구조의 field에 값을 저장한다.
- `INCR` : 숫자 값을 1 증가시킨다.
- `DEL` : Key와 데이터를 삭제한다.
- `EXPIRE` : Key의 만료 시간을 설정한다.

또한

- 트랜잭션 처리 : 여러 명령어를 하나의 작업 단위처럼 순차 실행한다.
- Lua Script 실행 : 여러 Redis 작업을 Lua 스크립트로 한 번에 처리한다. <br>
    (*) Lua 스크립트란? Redis 안에서 여러 명령어를 한 번에 실행할 수 있게 해주는 스크립트 기능
- Pub/Sub 처리 : 발행된 메시지를 구독 중인 클라이언트에게 전달한다.
- Pipeline 처리 : 여러 명령어를 한 번에 보내 네트워크 비용을 줄인다.

등도 메인 스레드에서 수행된다.

실제 데이터 변경과 조회는 모두 메인 스레드에서 처리된다.

#### b. 보조 스레드가 하는 일

보조 스레드는 메인 스레드의 부담을 줄이기 위한 작업들을 담당한다.

| 작업 | 설명 |
| --- | --- |
| Networking I/O | 클라이언트 read/write 처리 |
| AOF Rewrite | AOF 파일 재작성 |
| RDB 저장 | 스냅샷 생성 |
| 비동기 작업 | 백그라운드 작업 처리 |

Redis 6 이후에는 네트워크 입출력 작업 일부를 보조 스레드가 처리한다.

예시

- 클라이언트 요청 읽기(read)
- 응답 데이터 쓰기(write)

하지만 실제 Redis 명령 실행은 메인 스레드가 담당한다.

즉

- 보조 스레드 → 요청 읽기/응답 쓰기
- 메인 스레드 → Redis 명령 실행

구조이다.

또한

- RDB 저장
- AOF Rewrite

같은 작업도 백그라운드에서 수행된다.

그 결과 메인 스레드가 중단되지 않고 요청을 계속 처리할 수 있다.

#### c. 이벤트 루프

Redis는 이벤트 루프 기반으로 동작한다.

클라이언트 요청이 들어오면 이벤트 큐에 저장하고 순서대로 매우 빠르게 처리한다.

예시:

- GET
- SET
- HSET
- INCR

같은 명령들이 하나의 메인 스레드에서 순차적으로 실행된다.

#### d. 논블로킹 비동기 기반 동작

Redis는 단 하나의 메인 스레드만 사용하지만 블로킹 방식으로 동작하지 않는다.

논블로킹 비동기 방식으로 이벤트를 처리하면서 동시에 많은 요청을 처리할 수 있다.

특징

- 하나의 스레드가 요청을 기다리며 멈추지 않음
- 처리 가능한 작업부터 빠르게 수행
- 이벤트 루프 기반으로 요청들을 계속 순회하며 처리

이 구조 덕분에 높은 동시성을 제공할 수 있다.

#### e. Redis가 싱글 스레드를 사용하는 이유

멀티 스레드 환경에서는 여러 스레드가 동시에 데이터에 접근할 수 있다.

이 경우

- Lock
- synchronized
- 동기화 비용
- 데드락 위험

등이 발생할 수 있다.

Redis는 이러한 문제를 줄이기 위해 싱글 스레드 방식을 선택하였다.

그 결과

- 락이 필요 없음
- 데이터 정합성 유지 쉬움
- 구현 구조 단순
- 안정성 증가

#### f. Redis가 빠른 이유

싱글 스레드인데도 Redis가 빠른 이유는 다음과 같다.

- 메모리 기반 저장
- 명령어 구조가 단순
- 이벤트 루프 기반 처리
- 컨텍스트 스위칭 비용 감소
- Lock 사용 없음

또한 대부분의 Redis 명령은 매우 짧은 시간 안에 처리된다.

#### g. 최소 하나의 CPU 코어만 있어도 동작 가능

Redis는 메인 스레드 하나만으로도 핵심 기능을 수행할 수 있다.

그 결과:

- 최소 하나의 CPU 코어만 있어도 운영 가능
- 배포가 쉬움
- CPU 자원이 적은 환경에서도 높은 성능 제공

#### h. Redis 6.0 이후 변화

Redis 6 이후에는 일부 기능이 멀티 스레드로 동작한다.

Redis는 데이터 정합성을 위해 핵심 데이터 연산은 싱글 스레드로 유지하면서 네트워크 IO와 백그라운드 작업은 멀티 스레드로 최적화하였다.

멀티 스레드로 동작하는 부분

| 기능 | 설명 |
| --- | --- |
| Networking I/O | read/write 처리 |
| AOF Rewrite | 로그 파일 재작성 |
| RDB 저장 | 백그라운드 스냅샷 생성 |
| I/O 멀티스레딩 | 네트워크 병렬 처리 |

멀티 스레드의 이점

- I/O 병목 감소
- 네트워크 처리 최적화
- 메모리 절약
- TPS 증가 가능

싱글 스레드로 동작하는 부분

| 기능 | 설명 |
| --- | --- |
| GET/SET | 데이터 읽기/쓰기 |
| HSET | Hash 데이터 처리 |
| INCR | 카운트 증가 |
| EXPIRE/DEL | 키 공간 관리 |
| 트랜잭션 | MULTI/EXEC |
| Lua Script | EVAL |
| Pub/Sub | 메시징 |
| Pipeline 실행 | 명령 처리 |

#### i. 싱글 스레드의 단점

Redis는 하나의 명령어가 실행되는 동안 다른 요청들은 대기 상태가 된다.

따라서 실행 시간이 긴 명령어가 존재하면 병목이 발생할 수 있다.

대표적인 예시

- SORT
- SAVE
- KEYS *

이러한 명령어는 성능 저하를 유발할 수 있어 주의가 필요하다.

### (5) TTL(Time To Live)

Redis는 특정 시간이 지나면 데이터를 자동 삭제할 수 있다.

대표 사용 예시

- 로그인 세션
- 인증번호
- 임시 캐시 데이터

관련 명령어

- EXPIRE
- TTL
- SETEX

실무에서 매우 자주 사용되는 기능이다.

### (6) Cache 전략

Redis는 캐시 서버로 매우 많이 사용된다.

#### a. Cache Aside Pattern

가장 대표적인 캐시 전략이다.

동작 방식

1. 캐시 조회
2. 데이터 없으면 DB 조회
3. DB 결과를 Redis 저장
4. 이후 Redis에서 조회

특징

- 가장 일반적인 방식
- 애플리케이션이 캐시 관리

#### b. Write Through

데이터 저장 시:

- DB 저장
- Redis 저장

을 동시에 수행한다.

장점

- 캐시 최신 상태 유지

단점

- 쓰기 성능 감소 가능

#### c. Write Back(Write Behind)

먼저 Redis에 저장 후 나중에 DB 반영

장점

- 쓰기 성능 우수

단점

- 장애 시 데이터 유실 가능성 존재

Redis에서는 Hit Rate가 매우 중요하다.

#### d. 요즘 실무

특히 웹 백엔드에서는 거의 대부분 Cache Aside를 사용한다.

이유
- 구현 단순
- 장애 대응 쉬움
- Redis 장애 나도 DB는 살아있음
- 데이터 정합성 관리가 상대적으로 쉬움

반대로 Write Back은 Redis만 믿고 있다가 아직 DB 반영 전에 Redis 장애 발생 → 데이터 유실 가능문제가 있어서 금융/결제 같은 곳에서는 조심해서 사용한다.


### (7) Eviction(제거) 정책

메모리가 가득 찼을 때 어떤 데이터를 제거할지 결정하는 정책이다.

대표 정책

| 정책 | 설명 |
| --- | --- |
| noeviction | 삭제 안 함 |
| allkeys-lru | 가장 오래 안 쓴 데이터 삭제 |
| volatile-lru | TTL 있는 데이터 중 LRU 삭제 |
| allkeys-random | 랜덤 삭제 |
- LRU (Least Recently Used) : 가장 오랫동안 사용하지 않은 데이터 제거
- LFU (Least Frequently Used) : 가장 적게 사용된 데이터 제거

### (8) Atomic 연산

Redis는 단일 명령어를 원자적으로 처리한다.

대표 명령어

- INCR
- DECR
- SETNX

싱글 스레드 기반이기 때문에 멀티 스레드 환경에서도 안전하게 동작 가능하다.

### (9) Redis 분산 락

Redis는 분산 락 구현에도 자주 사용된다.

(*) 분산락 이란?
여러 서버가 동시에 같은 자원에 접근하지 못하도록 제어하는 락이다.
쉽게 말하면 여러 서버 중 지금은 딱 하나만 작업할 수 있도록 해주는 락이다.

#### a. 왜 필요한가?

여러 서버가 동시에 같은 자원 수정 시 문제 발생 가능

예시

- 중복 예약
- 중복 결제
- 재고 감소

#### b. Redis 분산 락 방식

대표 방식

- `SETNX` : Key가 없을 때만 값을 저장하는 Redis 명령어
- `EXPIRE` : 특정 Key의 만료 시간을 설정하는 Redis 명령어

또는

- Redisson : Redis 기반 분산 락 등을 쉽게 사용할 수 있게 해주는 Java 라이브러리
- RedLock : 여러 Redis 서버를 이용해 안정적으로 분산 락을 구현하는 알고리즘

같은 라이브러리 사용

#### c. Redis가 분산 락에 적합한 이유

- 싱글 스레드 기반
- 원자적 명령 처리 가능
- 매우 빠른 성능 제공

### (10) Pipeline

Pipeline은 여러 명령을 한 번에 전송하여 네트워크 비용을 줄이는 기능이다.

장점

- RTT 감소
- 성능 향상

하지만 실제 실행은 싱글 스레드에서 순차 처리된다.

### (11) Transaction

Redis도 트랜잭션 기능을 제공한다.

관련 명령어

- MULTI
- EXEC
- WATCH

하지만 RDBMS 트랜잭션과는 차이가 존재한다.

특징

- Rollback 개념 약함
- 완전한 ACID 보장은 아님

### (12) Pub/Sub

Redis는 메시지 브로커 기능도 제공한다.

동작 방식:

- Publisher가 메시지 발행
- Subscriber가 메시지 수신

사용 예시:

- 실시간 채팅
- 실시간 알림
- 이벤트 처리

### (13) Redis Stream

Redis Stream은 Pub/Sub보다 발전된 메시징 기능이다.

특징:

- 메시지 저장 가능
- Consumer Group 지원
- Queue 시스템 구현 가능

Kafka와 자주 비교된다.

### (14) 고가용성(HA)

Redis는 고가용성 기능을 제공한다.

복제를 통해 데이터를 여러 서버에 분산할 수 있다.

(*) 고가용성 이란?
장애가 나도 서비스가 멈추지 않게 만드는 구조이다.

#### a. Replication

Master 서버의 데이터를 Replica 서버로 복제 가능하다.

이를 통해 장애 발생 시 데이터 손실 위험을 줄일 수 있다.

#### b. Sentinel

Redis Sentinel은 장애 감지 및 자동 Failover 기능을 제공한다.

(*) Failover이란?
시스템에서 장애가 나면 대체 서버로 자동 전환되는 것이다.

기능

- 마스터 장애 감지
- 새로운 마스터 선출
- 자동 Failover 수행
- 서비스 자동 복구 지원

애플리케이션은 Sentinel을 통해 새로운 마스터를 자동으로 인식할 수 있다.

### (15) Redis Cluster

Redis는 Cluster 모드를 통해 수평 확장이 가능하다.

→ Cluster 모드는 별도 설정 + 구성 작업이 필요하다.

특징:

- 자동 샤딩
- 데이터 분산 저장
- 복제 지원
- 자동 Failover
- 클러스터 버스를 통한 노드 간 통신

애플리케이션은 데이터가 어느 샤드에 저장되어 있는지 직접 관리하지 않아도 된다.

### (16) 클라우드 네이티브

Redis는 클라우드 환경에 최적화되어 있다.

#### a. 클라우드 네이티브란?

클라우드 환경에 맞춰 애플리케이션을 개발·운영하는 방식을 의미한다.

포함되는 개념:

- 마이크로서비스
- 컨테이너
- 오케스트레이션
- DevOps

Redis는 빠른 데이터 처리 성능 덕분에 마이크로서비스 환경과 잘 어울린다.

#### b. 멀티 클라우드

여러 클라우드 서비스 제공업체를 함께 사용하는 전략이다.

예시

- AWS
- Google Cloud
- Azure

장점:

- 장애 위험 감소
- 특정 클라우드 의존성 감소
- 지역 분산 가능
- 지연 시간 감소

Redis는 멀티 클라우드 환경에서도 일관된 성능과 기능을 제공한다.

(*) 멀티 클라우드 vs 하이브리드 클라우드
- 멀티 클라우드 : 여러 클라우드 서비스를 함께 사용하는 구조
- 하이브리드 클라우드 : 온프레미스 + 클라우드

## 3. Redis의 장점

- 매우 빠른 성능
- 단순한 구조
- 다양한 자료구조 제공
- 높은 확장성
- 실시간 시스템에 적합

## 4. Redis의 단점

- 메모리 비용이 높음
- 긴 명령어에 취약
- 기본적으로 휘발성

## 5. Redis 사용 시 주의점

- KEYS * 사용 주의
    - 모든 키를 조회하는 작업은 성능 문제를 유발할 수 있다.
    - 실무에서는 SCAN(조금씩 나눠서 조회) 사용 권장
- Hot Key 문제
    - 특정 Key에 요청이 몰리면 병목 발생 가능
- 대용량 데이터 저장 주의
    - Redis는 메모리 기반이므로 메모리 비용 문제가 발생할 수 있다.
- TTL 없는 캐시 주의
    - TTL 없이 캐시를 저장하면 메모리 증가 문제 발생 가능

## 6. Redis와 RDBMS 차이

| Redis | RDBMS |
| --- | --- |
| 메모리 기반 | 디스크 기반 |
| 매우 빠름 | 상대적으로 느림 |
| 단순 조회 강점 | 복잡한 쿼리 강점 |
| 캐시 용도 많음 | 영구 저장 강점 |

## 7. 정리

Redis는 메모리 기반의 고성능 Key-Value 데이터 저장소이다.

핵심 데이터 처리는 싱글 스레드 이벤트 루프 기반으로 동작하며 락 없이 빠르고 안정적으로 요청을 처리한다.

또한 다양한 자료구조, Persistence, Cache 전략, 분산 락, Cluster, Sentinel 기능 등을 제공하여 대규모 실시간 서비스 환경에서 널리 사용된다.
